# 開発共通規約

## Git

### コミットメッセージ

- プレフィックスにgitmoji(https://gitmoji.dev/)を使用すること
- コミットメッセージは簡潔かつ明確にすること
- コミットメッセージに説明が必要な場合は、本文に詳細を記載すること

### Issue作成規約

#### Issueのタイトル

- タスクの内容が一目でわかるように簡潔かつ明確に記載すること
- 例: 「給油履歴個別取得APIの実装と編集画面の修正」

##### プレフィックス

- 新機能、PG修正: `[Feature] {タイトル}`
- バグ修正: `[Bug] {タイトル}`
- 作業タスク: `[Task] {タイトル}`

#### Issueの本文構成

以下のセクションを含めること：

##### 1. 概要

- タスクの背景と目的を簡潔に記載
- 何を改善・実装するのかを明確にする

##### 2. 現状の問題点（該当する場合）

- 現在の実装の問題点や課題を箇条書きで記載
- パフォーマンスやUXの問題があれば具体的な数値を含める

##### 3. 実装内容

- 実装するファイルと変更内容を明記
- APIエンドポイントの仕様（リクエスト/レスポンス）
- 主要なロジックの説明

##### 4. 検証方法

- ビルド・リント確認のコマンド
- 動作確認の手順
- パフォーマンス確認方法（該当する場合）

##### 5. 期待される改善（該当する場合）

- パフォーマンス向上の数値
- UX改善の内容
- その他の効果

#### ラベルの付与

以下のラベルを適切に付与すること：

- **種類**:
  - `enhancement`: 新機能・改善
  - `bug`: バグ修正
  - `documentation`: ドキュメント
- **対象**:
  - `Web`: Webプロジェクト
  - `Mobile`: モバイル
  - `LP`: ランディングページ
  - `Infra`: インフラ, CD/CD, AWS CDK, 設定タスク
- **その他**: 必要に応じて
  - `good first issue`
  - `help wanted`

#### Issue作成コマンド例

```bash
gh issue create \
  --title "タスクのタイトル" \
  --body "$(cat <<'EOF'
## 概要
...

## 現状の問題点
...

## 実装内容
...

## 検証方法
...

## 期待される改善
...
EOF
)" \
  --label "enhancement" \
  --label "Web"
```
