# Issue対応計画の立案

あなたはGitHub Issue #{issue_number}の対応計画を立案する専門家です。

## 実行手順

### 1. Issue詳細の取得

まず、`gh issue view {issue_number} --json title,body,labels,assignees,state | cat` コマンドを実行してIssueの詳細情報を取得してください。

### 2. コードベースの調査

Explore agentを使用して、以下の観点でコードベースを調査してください:

- Issue内容に関連するファイルやコンポーネントの特定
- 既存の類似実装の確認
- 影響を受ける可能性のあるモジュールの洗い出し
- 使用されている技術スタックやパターンの理解

### 3. 対応計画の作成

以下の4つのセクションを含む詳細な計画を日本語で作成してください:

#### 📋 タスク分解
実装に必要な具体的なタスクをチェックリスト形式で列挙してください。
- 各タスクは実行可能な粒度に分解
- 前後関係や依存関係を考慮した順序で記載
- 所要時間の目安も記載（オプション）

#### 🔍 技術調査項目
実装前に調査・確認が必要な技術的な項目をリストアップしてください。
- 使用する技術・ライブラリの選定理由
- 未確定の技術的課題
- パフォーマンスやセキュリティの考慮事項

#### 🛠️ 実装アプローチ
具体的な実装方法や技術選定の提案を記載してください。
- 採用する技術スタックや設計パターン
- ファイル構成やディレクトリ配置
- コーディング規約やベストプラクティスの適用

#### 📊 影響範囲分析
変更が影響を与える可能性のあるファイルやコンポーネントを特定してください。
- 修正対象のファイルリスト
- 影響を受ける可能性のある関連機能
- 必要なテストの範囲

### 4. プランの確認

作成した計画を整形されたMarkdown形式でユーザーに提示し、内容の確認を求めてください。

### 5. Issueへのコメント投稿

ユーザーが計画を承認したら、以下のコマンドでIssueにコメントとして投稿してください:

```bash
gh issue comment {issue_number} --body "$(cat <<'EOF'
# 📝 対応計画

[作成した計画の内容をここに貼り付け]

---
🤖 この計画は Claude Code の `/plan-issue` コマンドで自動生成されました
EOF
)"
```

## 注意事項

- すべてのコミュニケーションは日本語で行ってください
- `development/README.md` に記載された開発ルールに従ってください
- コードベースの既存パターンを尊重してください
- 実装の実現可能性を常に考慮してください
- Issue番号は `{issue_number}` として参照してください（スラッシュコマンドの引数から自動設定されます）
